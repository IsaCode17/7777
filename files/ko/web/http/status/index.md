{{HTTPSidebar}}

HTTP 응답 상태 코드는 특정 HTTP 요청이 성공적으로 완료되었는지 알려줍니다.
응답은 5개의 그룹으로 나누어집니다: 

1. [정보 응답](#정보_응답) (`100` – `199`)
2. [성공 응답](#성공_응답) (`200` – `299`)
3. [리다이렉션 메시지](#리다이렉션_메시지) (`300` – `399`)
4. [클라이언트 에러 응답](#클라이언트_에러_응답) (`400` – `499`)
5. [서버 에러 응답](#서버_에러_응답) (`500` – `599`)

상태 코드는 [RFC 9110](https://httpwg.org/specs/rfc9110.html#overview.of.status.codes)에 의해 정의되어 있습니다.

> **참고:** [이 목록](#information_responses)에 없는 상태 코드는 비표준이며 서버 소프트웨어에 사용자 지정되었을 수 있습니다.

## 정보 응답

- {{HTTPStatus(100, "100 Continue")}}
  - : 이 임시적인 응답은 지금까지의 상태가 괜찮으며 클라이언트가 계속해서 요청을 하거나 이미 요청을 완료한 경우에는 무시해도 되는 것을 알려줍니다.
- {{HTTPStatus(101, "101 Switching Protocols")}}
  - : 이 코드는 클라이언트가 보낸 {{HTTPHeader("Upgrade")}} 요청 헤더에 대한 응답에 들어가며 서버에서 프로토콜을 변경할 것임을 알려줍니다.
- {{HTTPStatus(102, "102 Processing")}} ({{Glossary("WebDAV")}})
  - : 이 코드는 서버가 요청을 수신하였으며 이를 처리하고 있지만, 아직 제대로 된 응답을 알려줄 수 없음을 알려줍니다.
- {{HTTPStatus(103, "103 Early Hints")}} {{experimental_inline}}
  - : 이 상태 코드는 주로 {{HTTPHeader("Link")}} 헤더와 함께 사용되어 서버가 응답을 준비하는 동안 사용자 에이전트가(user agent) 사전 로딩([preloading](/ko/docs/Web/HTML/Preloading_content))을 시작할 수 있도록 합니다.

## 성공 응답

- {{HTTPStatus(200, "200 OK")}}

  - : 요청이 성공적으로 처리 되었습니다. 성공의 의미는 HTTP 메소드에 따라 다릅니다:
    - `GET`: 리소스를 불러와서 메시지 본문에 전송되었습니다.
    - `HEAD`: 개체 해더가 메시지 본문에 있습니다.
    - `PUT` 또는 `POST`: 수행 결과에 대한 리소스가 메시지 본문에 전송되었습니다.
    - `TRACE`: 메시지 바디는 서버에서 수신한 요청 메시지를 포함하고 있습니다.

- {{HTTPStatus(201, "201 Created")}}
  - : 요청이 성공적이었으며 그 결과로 새로운 리소스가 생성되었습니다. 이 응답은 일반적으로 `POST` 요청 또는 일부 `PUT` 요청 이후에 따라옵니다.
- {{HTTPStatus(202, "202 Accepted")}}
  - : 요청을 수신하였지만 아직 그에 응하여 행동하지 않았습니다. 이 응답은 비확약적인데, HTTP에서는 추후 요청의 결과를 비동기적으로 전송할 수 있는 방법이 없기 때문입니다. 이것은 다른 프로세스 또는 서버가 요청을 다루고 있는 상황 또는 일괄 처리를 위해 만들어졌습니다.
- {{HTTPStatus(203, "203 Non-Authoritative Information")}}
  - : 이 응답 코드는 돌려받은 메타데이터 세트가 오리진 서버의 것과 일치하지 않지만 로컬이나 서드 파티 복사본에서 모아졌음을 의미합니다. 이 응답은 미러 또는 다른 리소스의 백업본에서 주로 사용됩니다. 이러한 경우를 제외하면, `200 OK` 응답이 더 선호됩니다.
- {{HTTPStatus(204, "204 No Content")}}
  - : 요청에 대해서 보내줄 수 있는 콘텐츠가 없지만, 헤더는 의미있을 수 있습니다. 사용자 에이전트는 해당 리소스에 대해 캐시된 헤더를 새로운 것으로 업데이트 할 수 있습니다.
- {{HTTPStatus(205, "205 Reset Content")}}
  - : 이 응답 코드는 요청을 완수한 이후에 사용자 에이전트에게 이 요청을 보낸 문서 뷰를 리셋하라고 알려줍니다.
- {{HTTPStatus(206, "206 Partial Content")}}
  - : 이 응답 코드는 클라이언트에서 리소스의 일부분만 요청하기 위해 {{HTTPHeader("Range")}} 헤더를 전송하였을 때 사용됩니다.
- {{HTTPStatus(207, "207 Multi-Status")}} ({{Glossary("WebDAV")}})
  - : 여러 상태 코드가 적절할 수 있는 상황에서 여러 리소스에 대한 정보를 전달합니다.
- {{HTTPStatus(208, "208 Already Reported")}} ({{Glossary("WebDAV")}})
  - : `<dav:propstat>` 응답 속성에서 사용되며, 동일 컬렉션에서 내부 멤버를 반복적으로 열거하는 것을 피하기 위해 사용됩니다.
- {{HTTPStatus(226, "226 IM Used")}} ([HTTP Delta encoding](https://datatracker.ietf.org/doc/html/rfc3229))
  - : 서버가 GET 요청에 대한 리소스의 의무를 다 했으며, 응답은 하나 또는 그 이상의 인스턴스 조작이 현재 인스턴스에 적용이 되었음을 알려줍니다.

## 리다이렉션 메시지

- {{HTTPStatus(300, "300 Multiple Choices")}}
  - : 요청에 대해서 하나 이상의 응답이 가능합니다. 사용자 에이전트 또는 사용자는 그중에 하나를 반드시 선택해야 합니다. (응답 중 하나를 선택하는 방법에 대한 표준화 된 방법은 존재하지 않지만, 사용자가 선택할 수 있게 선택지에 대한 HTML 링크가 권장됩니다.)
- {{HTTPStatus(301, "301 Moved Permanently")}}
  - : 이 응답 코드는 요청한 리소스의 URL이 영구적으로 변경되었음을 의미합니다. 응답에서 새로운 URL이 주어집니다.
- {{HTTPStatus(302, "302 Found")}}
  - : 이 응답 코드는 요청한 리소스의 URI가 일시적으로 변경되었음을 의미합니다. URI는 나중에 변경될 수 있습니다. 그러므로, 클라이언트는 향후의 요청도 반드시 동일한 URI로 해야합니다.
- {{HTTPStatus(303, "303 See Other")}}
  - : 서버가 클라이언트에게 클라이언트가 요청한 리소스를 다른 URI에서 GET 요청을 통해 얻어야 함을 지시하기 위해 보내는 응답입니다.
- {{HTTPStatus(304, "304 Not Modified")}}
  - : 이것은 캐시를 목적으로 사용됩니다. 이것은 클라이언트에게 응답이 수정되지 않았음을 알려주며, 그러므로 클라이언트는 계속해서 응답의 캐시된 버전을 사용할 수 있습니다.
- `305 Use Proxy` {{deprecated_inline}}
  - : 이전 버전의 HTTP 기술 사양에서 정의되었으며, 요청한 응답은 반드시 프록시를 통해서 접속해야 하는 것을 알려줍니다. 이것은 프록시의 in-band 설정에 대한 보안상의 문제로 인하여 더 이상 사용되지 않습니다.
- `306 unused`
  - : 이 응답 코드는 더 이상 사용되지 않으며, 현재는 추후 사용을 위해 예약되어 있습니다. 이것은 HTTP/1.1 기술 사양의 이전 버전에서 사용되었습니다.
- {{HTTPStatus(307, "307 Temporary Redirect")}}
  - :  서버가 클라이언트에게 클라이언트가 요청한 리소스가 다른 URI에 있으며, 이전 요청과 동일한 메소드를 사용하여 요청해야함을 지시하기 위해 이 응답을 보냅니다. 이것은 `302 Found` HTTP 응답 코드와 동일한 의미를 가지고 있으며, 사용자 에이전트가 반드시 사용된 HTTP 메소드를 변경하지 말아야 하는 점만 다릅니다. 만약 첫 요청에 `POST`가 사용되었다면, 두번째 요청도 반드시 `POST`를 사용해야 합니다.
- {{HTTPStatus(308, "308 Permanent Redirect")}}
  - : 이것은 리소스가 이제 HTTP 응답 헤더의 `Location:` 에 명시된 영구히 다른 URI에 위치하고 있음을 의미합니다. 이것은 `301 Moved Permanently` HTTP 응답 코드와 동일한 의미를 가지고 있으며, 사용자 에이전트가 반드시 HTTP 메소드를 변경하지 말아야 하는 점만 다릅니다. 만약 첫 요청에 `POST`가 사용되었다면, 두번째 요청도 반드시 `POST`를 사용해야 합니다.

## 클라이언트 에러 응답

- {{HTTPStatus(400, "400 Bad Request")}}
  - : 서버가 클라이언트의 문제로 간주되는 어떠한 이유(예: 잘못된 응답 구문, 올바르지 않은 응답 메시지 프레이밍, 거짓된 응답 라우팅)로 인해 요청을 처리할 수 없거나 처리 하지 않았음을 의미합니다.
- {{HTTPStatus(401, "401 Unauthorized")}}
  - : 비록 HTTP 표준에서는 "허가 되지 않음(unauthorized)"를 명확히 하고 있지만, 의미상 이 응답은 "인증 되지 않음(unauthenticated)"을 의미합니다.
    클라이언트는 요청한 응답을 받기 위해서는 반드시 스스로를 인증해야 합니다.
- {{HTTPStatus(402, "402 Payment Required")}} {{experimental_inline}}
  - : 이 응답 코드는 나중에 사용될 것을 대비해 예약되었습니다.
     초기 목적은 디지털 결제 시스템에 사용하기 위함이였으나, 이 상태 코드는 매우 드물게 사용되며 표준 규약 또한 존재하지 않습니다.
- {{HTTPStatus(403, "403 Forbidden")}}
  - : 클라이언트는 콘텐츠에 접근할 권리를 가지고 있지 않습니다. 즉, 접근이 허가되지 않아 서버가 요청한 응답을 주기를 거부하고 있습니다. `401 Unauthorized`과 달리 서버는 클라이언트가 누구인지 알고 있습니다.
- {{HTTPStatus(404, "404 Not Found")}}
  - : 서버는 요청받은 리소스를 찾을 수 없습니다. 브라우저에서는 알려지지 않은 URL을 의미합니다. API에서는 추가로 끝점은 적절하지만 리소스 자체는 존재하지 않음을 의미하기도 합니다. 서버들은 인증받지 않은 클라이언트로부터 리소스를 숨기기 위하여 이 응답을 `403 Forbidden` 대신에 전송할 수도 있습니다. 이 응답 코드는 웹에서 자주 접할 수 있기 때문에 가장 유명할지도 모릅니다.
- {{HTTPStatus(405, "405 Method Not Allowed")}}
  - : 서버에서 요청한 메소드를 알고 있지만, 해당 리소스에 대해서 지원되지 않음을 의미합니다. 예를 들어, 어떤 API에서 리소스를 삭제하기 위해 `DELETE`를 사용하는 것을 금지할 수 있습니다.
- {{HTTPStatus(406, "406 Not Acceptable")}}
  - : 이 응답은 서버가 [서버 주도 콘텐츠 협상](ko/docs/Web/HTTP/Content_negotiation#%EC%84%9C%EB%B2%84_%EC%A3%BC%EB%8F%84_%EC%BD%98%ED%85%90%EC%B8%A0_%ED%98%91%EC%83%81) 을 수행한 이후, 사용자 에이전트에서 정해준 규격에 따른 어떠한 콘텐츠도 찾지 않았을 때, 웹 서버가 보냅니다.
- {{HTTPStatus(407, "407 Proxy Authentication Required")}}
  - : `401 Unauthorized`과 비슷하지만 프록시를 통한 인증이 필요합니다.
- {{HTTPStatus(408, "408 Request Timeout")}}
  - : 이 응답은 오랫동안 요청이 없는 연결에 일부 서버가 전송하며, 이전에 클라이언트로부터 어떠한 요청이 없었다고 하더라도 보내지기도 합니다. 이는 서버가 사용되지 않는 연결을 끊고 싶어한다는 것을 의미합니다. 이 응답은 최근 들어 더 자주 사용되는데, Chrome, Firefox 27+, 또는 IE9와 같은 일부 브라우저는 웹서핑 속도를 올리기 위해 HTTP 사전 연결 메커니즘을 사용하기 때문입니다. 일부 서버는 이 메시지를 보내지 않고 연결을 끊어버리기도 한다는 점도 참고해야 합니다.
- {{HTTPStatus(409, "409 Conflict")}}
  - : 이 응답은 요청이 현재 서버의 상태와 충돌될 때 보냅니다.
- {{HTTPStatus(410, "410 Gone")}}
  - : 이 응답은 요청한 콘텐츠가 서버에서 영구적으로 삭제되었으며, 전달해 줄 수 있는 주소 역시 존재하지 않을 때 보냅니다. 클라이언트는 리소스에 대한 링크와 캐시를 삭제하기를 기대합니다. HTTP 기술 사양은 이 상태 코드가 "일시적인, 홍보용 서비스"에 사용되기를 의도니다. API는 삭제된 리소스에 대해 이 코드를 억지로 사용할 필요는 없습니다.
- {{HTTPStatus(411, "411 Length Required")}}
  - : 서버에서 `Content-Length` 헤더 필드를 필요로 하지만 이를 정의하지 않은 요청이 들어왔기 때문에 서버가 요청을 거절했습니다.
- {{HTTPStatus(412, "412 Precondition Failed")}}
  - : 클라이언트가 서버와 맞지 않는 Precondition을 헤더에 명시했습니다.
- {{HTTPStatus(413, "413 Payload Too Large")}}
  - : 요청 엔티티는 서버에서 정의한 한계보다 큽니다. 서버는 연결을 끊거나 `Retry-After` 헤더 필드를 반환할 수 있습니다.
- {{HTTPStatus(414, "414 URI Too Long")}}
  - : 클라이언트가 요청한 URI는 서버에서 처리할 수 있는 길이보다 깁니다.
- {{HTTPStatus(415, "415 Unsupported Media Type")}}
  - : 요청한 미디어 포맷을 서버에서 지원하지 않아 서버가 해당 요청을 거절했습니다.
- {{HTTPStatus(416, "416 Requested Range Not Satisfiable")}}
  - : `Range` 헤더 필드에 지정한 범위를 만족시킬 수 없습니다. 범위가 타겟 URI 데이터의 크기를 벗어났을 가능성이 있습니다.
- {{HTTPStatus(417, "417 Expectation Failed")}}
  - : 이 응답 코드는 `Expect` 요청 헤더 필드로 요청한 예상을 서버가 만족시킬 수 없음을 알려줍니다.
- {{HTTPStatus(418, "418 I'm a teapot")}}
  - : 서버는 커피를 찻 주전자에 끓이는 것을 거절합니다.
- {{HTTPStatus(421, "421 Misdirected Request")}}
  - : 응답을 생성할 수 없는 서버로 요청이 전송되었습니다. 이 응답은 요청 URI에 포함된 도식과 권한이 조합을 처리할 수 있게 구성되지 않은 서버에서 전송될 수 있습니다.
- {{HTTPStatus(422, "422 Unprocessable Entity")}} ({{Glossary("WebDAV")}})
  - : 요청은 잘 구성되었으나, 문법 오류로 인하여 따를 수 없습니다.
- {{HTTPStatus(423, "423 Locked")}} ({{Glossary("WebDAV")}})
  - : 접근하려고 하는 리소스는 잠겨있습니다.
- {{HTTPStatus(424, "424 Failed Dependency")}} ({{Glossary("WebDAV")}})
  - : 이전 요청이 실패하였기 때문에 지금의 요청도 실패하였습니다.
- {{HTTPStatus(425, "425 Too Early")}} {{experimental_inline}}
  - : 서버가 재전송 될 수 있는 요청을 처리하는 데 위험을 감수하기를 원하지 않음을 의미합니다.
- {{HTTPStatus(426, "426 Upgrade Required")}}
  - : 서버가 현재 프로토콜을 사용하여 요청을 처리하는 것은 거부하지만, 클라이언트가 다른 프로토콜로 업그레이드한 후에는 요청을 수행할 의향이 있음을 나타냅니다. 서버는 {{HTTPHeader("Upgrade")}} 헤더와 필요로 하는 프로토콜을 426 응답에 같이 보냅니다.
- {{HTTPStatus(428, "428 Precondition Required")}}
  - : 오리진 서버에서 조건적인 요청을 필요로 합니다. 클라이언트가 리소스를 `GET`해서, 수정하고, 그리고 `PUT`으로 서버에 돌려놓는 동안 서드 파티가 서버의 상태를 수정하여 발생하는 충돌인 '업데이트 상실'을 예방하기 위한 목적입니다.
- {{HTTPStatus(429, "429 Too Many Requests")}}
  - : 사용자가 지정된 시간에 너무 많은 요청을 보냈습니다("rate limiting").
- {{HTTPStatus(431, "431 Request Header Fields Too Large")}}
  - : 요청한 헤더 필드가 너무 크기 때문에 서버는 요청을 처리하지 않을 것입니다. 요청은 크기를 줄인 다음에 다시 전송해야 합니다.
- {{HTTPStatus(451, "451 Unavailable For Legal Reasons")}}
  - : 사용자 에이전트가 정부에 의해 검열된 웹 페이지와 같이 합법적으로 제공할 수 없는 리소스를 요청했습니다.

## 서버 에러 응답

- {{HTTPStatus(500, "500 Internal Server Error")}}
  - : 서버에서 예상하지 못한 상황이 발생했습니다.
- {{HTTPStatus(501, "501 Not Implemented")}}
  - : 요청한 메소드를 서버에서 지원되지 않아 처리할 수 없습니다. 서버가 의무적으로 지원해야 하는(즉, 이 코드를 반환해서는 안되는) 메소드는 `GET` 과 `HEAD` 뿐입니다.
- {{HTTPStatus(502, "502 Bad Gateway")}}
  - : 이 오류 응답은 서버가 요청을 처리하는 데 필요한 응답을 얻기 위해 게이트웨이로서 작업하는 동안 잘못된 응답을 수신했음을 의미합니다.
- {{HTTPStatus(503, "503 Service Unavailable")}}
  - : 서버가 요청을 처리할 준비가 되지 않았습니다. 일반적인 원인은 유지보수를 위해 작동이 중단되거나 과부하가 걸렸을 때 입니다. 이 응답과 함께 문제를 설명하는 사용자 친화적인 페이지가 전송되어야 한다는 점에 유의하십시오. 이 응답은 일시적인 상황에 사용되어야 하며, `Retry-After` HTTP 헤더는 가능하면 서비스를 복구하기 전 예상 시간을 포함해야 합니다. 이러한 일시적인 상황을 나타내는 응답은 일반적으로 캐시되지 않아야 하므로 웹마스터는 이 응답과 함께 전송되는 캐싱 관련 헤더에 대해서도 주의해야 합니다.
- {{HTTPStatus(504, "504 Gateway Timeout")}}
  - : 이 오류 응답은 서버가 게이트웨이 역할을 하는 중 적시에 응답을 받지 못했을 때 주어집니다.
- {{HTTPStatus(505, "505 HTTP Version Not Supported")}}
  - : 요청에 사용된 HTTP 버전은 서버에서 지원되지 않습니다.
- {{HTTPStatus(506, "506 Variant Also Negotiates")}}
  - : 서버에 내부 구성 오류가 있습니다. 선택된 가변 리소스는 자연스러운 콘텐츠 협상 자체를 시작하도록 구성되어 있습니다. 즉, 협상 과정에 있어 적절한 끝점이 아닙니다.
- {{HTTPStatus(507, "507 Insufficient Storage")}} ({{Glossary("WebDAV")}}
  - : 서버가 요청을 정상적으로 처리하기 위한 표현을 저장하지 못해 메소드를 수행할 수 없음을 뜻합니다.
- {{HTTPStatus(508, "508 Loop Detected")}} ({{Glossary("WebDAV")}})
  - : 서버가 요청을 처리하는 동안 무한 루프를 감지했습니다.
- {{HTTPStatus(510, "510 Not Extended")}}
  - : 서버가 요청을 이행하려면 요청에 대한 추가 확장이 필요합니다.
- {{HTTPStatus(511, "511 Network Authentication Required")}}
  - : 클라이언트가 네트워크 액세스를 얻기 위해 인증을 받아야 할 필요가 있음을 나타냅니다.
