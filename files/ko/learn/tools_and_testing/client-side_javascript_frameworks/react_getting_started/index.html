이 페이지에서는 React를 시작하는 방법을 설명하겠습니다. React의 배경기술과 사용방법에 대해 설명하며 로컬환경에서 기본적 React 툴을 설정합니다. 또한 심플한 앱을 통해 React가 어떤 프로세스로 기능하고 있는가 배워갑니다.

전제조건:

HTML、CSS、JavaScript 에 익숙해야합니다.터미널/커맨드라인 에 관한 지식이 있어야 합니다.。

React는 JSX(JavaScript와 XML)를 사용합니다. 이는 HTML-in-JavaScript로 받아들이실 수 있습니다. HTML과 JavaScript 익숙하다면 JSX 학습이 보다 수월할 것입니다. 또한 어플리케이션에서 일어나는 버그가 JavaScript에 관한 것인가 React에 관한 것인가 특정하는 데에도 도움이 될 것 입니다.

목적: 로컬 상태의 React 개발환경을 설정하여 윤곽을 작성합니다. 그리고 이것이 어떠한 기능을 하는가 이해합니다.

React 의 기초

공식 캐치프레이즈에서 밝혔듯, React 는 UI를 구축하기 위한 라이브러리입니다. React는 프레임워크가 아니며, 웹에서만 작동하는 것도 아닙니다. 다른 라이브러리와 함께 사용하면서, 특정 환경에서 렌더링을 합니다. 예를 들어 React Native 는 모바일 어플리케이션 구축에 사용됩니다. React 360 을 사용하면, 가상현실 어플리케이션을 구축할 수 있습니다. 이 외에도 많은 가능성을 지니고 있습니다.

웹 구축을 위해 개발자들은 ReactDOM 와 React를 사용합니다. React와 ReactDOM은 다른 웹 개발 프레임워크와 같은 선상에서 논의되기에 우리는 React가 프레임워크라 일컬어질 때는 우리는 이것이 무엇을 의미하는 것인지 이해해야 합니다.

React 가 지향하는 것은, 개발자가 UI 구축을 할 때 발생하는 버그를 최소한으로 억누르는 것입니다. 이는 컴포넌트를 사용함으로 이루어집니다. (사용자 인터페이스의 일부를 설명하는 자체 포함 논리 코드 조각입니다. 이러한 구성 요소는 함께 구성되어 전체 UI를 만들 수 있으며 렌더링 작업의 많은 부분을 추상화하여 UI 설계에 집중할 수 있습니다.)

유스 케이스

다른 프레임 워크와 달리 React는 코드 규칙이나 파일 구성에 대한 엄격한 규칙을 시행하지 않습니다. 이를 통해 팀들은 자신에게 가장 잘 맞는 협약을 설정하고, 그들이 원하는 어떤 방법으로든 반응을 채택할 수 있습니다. React는 버튼 하나를 다루거나, 인터페이스 몇 개 또는 앱의 전체 사용자 인터페이스를 처리할 수 있다.

React는 인터페이스의 일부에 사용할 수 있습니다. 그러나 jQuery 같은 라이브러리와 Vue 같은 프레임워크 어플리케이션에 녹아드는 것은 쉽지 않습니다. 오히려 React로 어플리케이션 전체를 구축하는 것이 더욱 접근하기 쉽습니다.

더욱이 JSX의 인터페이스를 작성 같이 React 어플리케이션 개발자 경험 상의 컴파일 프로세스가 필요합니다. Babel 컴파일러 같은 웹사이트를 추가하면 코드 실행은 느려지기 때문에 개발자는 빌드 과정에서 그것을 처리하는 설정하는 경우가 많습니다. 확실히 React는 꽤나 많은 툴들을 요구하지만 우리는 반드시 배울 수 있습니다.

이 글에서는 Facebook의 create-react-app 를 사용하여 어플리케이션 인터페이스를 구축해 나갈 것입니다.

React에 의한 JavaScript 작성 방법

React는 패턴이 다양한 최신 JavaScript 기능을 이용하고 있습니다. JavaScript와 차이나는 가장 큰 부분은 JSX 구문을 사용하는 것입니다. JSX는 JavaScript 구문을 확장하여 HTML 같은 코드를 작성할 수 있습니다. 예를 들어: 

const heading = <h1>Mozilla Developer Network </h1>;

이러한 상수 heading은 JSX 표현식이라 불려집니다. React는 이 처럼 <h1> 태그를 렌더링할 수 있습니다.

의미론적으로, <header> 태그를 감싸고 싶을 때, JSX에서는 HTML처럼 필요한 요소를 포함시키면 됩니다. 

const header = (
  <header>
    <h1>Mozilla Developer Network</h1>
  </header>
);

Note: 이 스니펫에서 괄호는 JSX 고유한 것이 아닙니다. 어플리케이션에 영향을 주지 않습니다. 여러 줄의 코드가 같은 표현이라는 뜻입니다. 아래와 같이 header를 정의하는 것이 좋을 것입니다. 

const header = <header>
    <h1>Mozilla Developer Network</h1>
</header>

다만,<header> 이러한 표현을 시작하는 태그와 닫힘 태그가 동일한 위치에 있지 않기에 이런 표현이 어색해 보일수도 있겠네요.

물론 브라우저는 저 상태의 JSX를 읽을 수 없습니다. (Babel 이나 Parcel 같은 툴을 사용해서) 컴파일 하면 header 식은 아래와 같이 됩니다. 

const header = React.createElement("header"、null、
  React.createElement("h1"、null、 "Mozilla Developer Network")
);

컴파일 단계를 건너뛰어 React.createElement() 를 사용해 직접 UI를 작성할 수도 있지만, 이러면 JSX가 가지고 있는 선언적 코드라는 장점은 사라지면서 코드를 읽기도 어려워집니다. 컴파일은 개발과정에서 추가적인 단계입니다만, React 커뮤니티의 많은 개발자들은 JSX가 가지고 있는 읽기 편함에 가치가 있다 여기고 있습니다. 더욱이 널리 사용되는 툴들은 JSX to JavaScript 편집을 설정 프로세스의 일부로 만듭니다. 우리는 원하는 경우가 아니면 스스로 그 프로세스를 구성하지 않아도 됩니다.

JSX는 HTML과 JavaScript가 합쳐진 것이므로, 개발자들은 직감적으로 사용할 수 있습니다. 하지만 그렇지 않은 사용자에게는 이러한 형태가 혼란스럽다 말할 수 있겠죠. 하지만 익숙해진다면, UI를 보다 빠르고 직관적으로 구축하면서 다른 사용자가 사용자의 코드베이스를 한 눈에 더 잘 이해할 수 있습니다.

JSX에 관해서 자세한 사항은 React 팀의 JSX In Depth 를 읽어보시길 바랍니다.

React 어플리케이션 준비하기

React 를 사용하는 방법은 다양합니다만, 이번에는 CLI(커맨드라인 인터페이스)에서 create-react-app를 사용합니다. 이를 통해 몇 가지의 패키지를 설치, 몇 가지 툴링을 처리하여 파일을 생성을 함으로 React 어플리케이션을 보다 신속히 개발할 수 있습니다.

일부 <script>요소를 HTML에 복사하여 create-react-app 없이 웹사이트에 추가할 수 있지만 create-react-app CLI는 React 애플리케이션 개발의 일반적인 시작점입니다. 이 프로그램을 사용해서 프로그램을 만든다면 더 많은 시간을 개발에 사용할 수 있으며 설치와 설정문제를 해결하는 데 더 적은 시간을 할애할 수 있습니다.

필요조건

create-react-app를 사용하기하기 위해 Node.js를 설치할 필요가 있습니다. 장기 지원 버전(LTS)를 사용하는 것을 권장합니다. Node에는 npm(노드 패키지 매니저), npx(노드 패키지 러너)가 포함되어 있습니다.

또한 npm 대신 Yarn 패키지매니저를 사용할 수도 있습니다만 일단 현재 단계에서는 npm 사용해봅시다. npm와 yarn에 대해 상세한 정보는 패키지 관리 기본를 참조하실 수 있습니다.

Windows를 사용하는 경우 이 튜토리얼에서 언급하고 있는 커맨드를 사용하기 위해서는 Unix / macOS 터미널과 같은 소프트웨어를 설치할 필요가 있습니다. Gitbash (Windows 용 git 툴을 제공) 혹은 Linux용 Windows 하위 시스템 (WSL) 둘 중 무엇이든 좋습니다. 혹시 일반적인 커맨드를 확인해보시려면 の両方が適しています。これら、および一般的な端末コマンドの詳細については、커멘드라인 크래시 코스를 참조해 주시길 바랍니다.

또한 React와 ReactDOM은 모던한 브라우저(IE9 이후)에서만 작동하기에 주의해주시길 바랍니다. 이 튜토리얼을 실행하기 위해서 Firefox, Safari, Chrome 등의 최신 브라우저를 사용하는 것을 권장합니다.

자세한 사항은 이하를 참조하시길 바랍니다.

  • nodejs.org (npm이란?)
  • npm 블로그 "npx 소개"
  • create-react-app 참조 문서

어플리케이션 초기화

create-react-app은 인수를 하나 받습니다. 바로 어플리케이션의 이름입니다. create-react-app은 이 이름을 사용한 디렉토리를 작성하여 필요한 파일을 생성합니다. cd를 통해 이동한 후, 터미널에 다음과 같은 커맨드를 실행합시다. 

npx create-react-app moz-todo-react

이는 moz-todo-react 디렉토리를 생성해, 그 안에서 몇 가지 일을 합니다:

  • 어플리케이션 기능에 반드시 필요한 npm 패키지를 설치합니다.
  • 어플리케이션을 기동하고 제공하기 위한 스크립트를 작성합니다.
  • 기본적인 어플리케이션 아키텍처를 정의하는 파일과 디렉토리 구조를 만듭니다.
  • 컴퓨터에 git이 설치되어 있는 경우, 디렉토리를 git 저장소로 초기화합니다.

Note: Yarn 패키지 매니저를 설치한 상태라면 create-react-app에서 npm 대신 사용됩니다. 명시적으로 npm를 사용하고 싶다면 create-react-app를 실행할 때 --use-npm 플러그를 추가하면 됩니다. 

npx create-react-app moz-todo-react --use-npm

create-react-app가 작동하는 동안, 몇 가지 메세지를 확인할 수 있습니다만, 정상 작동입니다! 여기에는 몇 분정도 걸릴 수 있으니 느긋하게 차라도 한 잔 하면서 기다려봅시다.

프로세스가 완료되었다면, moz-todo-react디렉토리로 cd 합시다. 그리고 커맨드에 npm start를 실행해 봅시다. create-react-app로 설치한 스크립트는, localhost:3000 로컬 서버를 열어, 새 브라우저 탭에 어플리케이션을 엽니다. 브라우저에는 다음과 같은 게시됩니다.

어플리케이션 구조

create-react-app은 React 어플리케이션을 개발에 필요한 모든 것을 제공합니다. 초기 파일구조는 다음과 같습니다. 

moz-todo-react
├──README.md
├──node_modules
├──package.json
├──package-lock.json
├──.gitignore
├──public
│├──favicon.ico
│├──index.html
│└──manifest.json
└──src
    ├──App.css
    ├──App.js
    ├──App.test.js
    ├──index.css
    ├──index.js
    ├──logo.svg
    └──serviceWorker.js

src 디렉토리는 어플리케이션 소스 코드가 있으면, 우리는 이곳을 긴 시간 다루게 될 것입니다.

public 디렉토리에는 어플리케이션 개발하는 동안 브라우저에서 읽을 파일이 포함되어 있습니다. 이 중에서 가장 중요한 것은index.html 입니다. React는 이 파일에 코드를 집어넣어, 브라우저에서 실행할 수 있도록 합니다. create-react-app 도움이 되는 다른 마크업도 몇 있으나, 여러분이 무엇을 하는지 알지 못하는 상태에서 편집하지 않도록 주의하세요. 어플리케이션에서 제목을 바꾸기 위해서 <title>에서 몇 가지 변경해야합니다. 접근성을 위해서 정확한 페이지 타이틀은 중요하기 때문이죠!

public 디렉토리는 어플리케이션 빌드하여 디플로이 할 때 공개됩니다. 이 튜토리얼에서는 디플로이 자체는 설명하지는 않겠습니다만, 어플리케이션 디플로이 튜토리얼에 설명한 유사한 솔루션을 확인하실 수 있습니다.

package.json 파일은 Node.js / npm가 프로젝트 정리를 위한 정보가 포함되어 있습니다. 이 파일은 React 어플리케이션 고유의 파일이 아닙니다. create-react-app가 이를 생성합니다. 이 튜토리얼을 완료하기 위해 이 파일을 이해하실 필요는 전혀 없습니다만, 보다 많이 알고 싶은 경우 "package.json은 무엇인가?"를 참조하시길 바랍니다. 또한 우리는 패키지 매니저 기본에서도 이야기 한 적 있습니다.

첫 React 컴포넌트 탐험하기 <App/>

React에서 컴포넌트는 어플리케이션의 일부로써 렌더링을 할 수 있는 재사용 가능한 모듈을 지칭합니다. 이러한 부품은 크거나 작아도 괜찮습니다만 보통 뚜렷한 하나의 목적이나 임무를 명확하게 정의합니다.

src/App.js를 열어 편집해봅시다. 이 파일은 우리의 첫 컴포넌트입니다. App에는 몇 행의 코드가 있습니다. 

import React from 'react';
import logo from './logo.svg';
import './App.css';

function App() {
  return (
    <div className="App">
      <header className="App-header">
        <img src={logo} className="App-logo" alt="logo" />
        <p>
          Edit <code>src/App.js</code> and save to reload.
        </p>
        <a
          className="App-link"
          href="https://reactjs.org"
          target="_blank"
          rel="noopener noreferrer"
        >
          Learn React
        </a>
      </header>
    </div>
  );
}
export default App;

App.js 파일은 중요한 3가지 부품으로 구성되어 있습니다. 상단에서 몇 가지 import문, 가운데 App 컴포넌트, 하단의 export입니다. 대부분 React 컴포넌트는 이러한 구성입니다.

import 문에 대해서

import는 파일 최상단에 선언되어, App.js에서 코드를 사용하는 것을 허용합니다. 이것을 자세히 봅시다. 

import React from 'react';
import logo from './logo.svg';
import './App.css';

첫 import는 React 라이브러리 자체를 import합니다. React는 작성한 JSX를 React.createElement()으로 변환하기 위해 모든 React 컴포넌트가 React 모듈을 import 해야 합니다. 여기를 지나친다면 어플리케이션에서 에러가 발생합니다.

두 번째 import는 './logo.svg'를 import 합니다. 맨 앞에 ./, 경로 뒤에 붙은 확장자 .svg에 주의해 주십시오. 이는 로컬 파일이며, JavaScript가 아님을 알려줍니다. 실제로 logo.svg 파일은 src/ 디렉토리에 있습니다.

React 모듈을 import 할 때 경로나 확장자를 적지 않습니다. — 대신 로컬 파일이 아니라 package.json에 의존적으로 나열되기 때문입니다. 이 수업을 통해서 이 차이를 신경써보시길 바랍니다.

3번째 import는 App 컴포넌트에 관련된 CSS를 import 합니다. 변수명, 지시어는 없습니다. 이 특별한 import 구문은 JavaScript만의 모듈은 아닙니다. Webpack이 제공하는 것입니다. Webpack에서는 모든 JavaScript 파일을 번들로 묶어 브라우저에 제공합니다.

App 컴포넌트

import 다음으로는 App 함수가 있습니다. 대부분의 JavaScript 커뮤니티는 helloWorld 같이 카멜케이스를 선호합니다만, React 컴포넌트는 HelloWorld처럼 파스칼케이스를 변수명으로 사용하여, JSX요소가 React 컴포넌트면서 HTML태그가 아님을 밝힙니다. App 함수명을 app으로 바꾸면 브라우저는 에러를 보여줍니다.

어플리케이션을 더욱 자세히 봅시다.

function App() {
  return (
    <div className="App">
      <header className="App-header">
        <img src={logo} className="App-logo" alt="logo" />
        <p>
          Edit <code>src/App.js</code> and save to reload.
        </p>
        <a
          className="App-link"
          href="https://reactjs.org"
          target="_blank"
          rel="noopener noreferrer"
        >
          Learn React
        </a>
      </header>
    </div>
  );
}

この App 関数は JSX 式を返します。この式には、ブラウザーが最終的に DOM にレンダリングするものを定義します。

HTML と同じように、attribute="value" のように属性をもつ要素があります。3 行目では、<div> 開始タグに className 属性があります。これは HTML のclass 属性と同じですが、JSX は JavaScript であるため使用できません。class が予約されているためです。つまり、JavaScript ではすでにそれを使用しており、もし同様に使うと問題が発生するということです。同じような理由で、他のいくつかの HTML 属性は、HTML とは異なる方法で JSX で記述されています。

6 行目の <p> タグを「Hello, world！」に変更して、ファイルを保存します。この変更はブラウザー http://localhost:3000 で実行されている開発サーバーですぐにレンダリングされます。次に、<a> タグを削除して保存します。すると、「Learn React」リンクはなくなります。

あなたの App コンポーネントは次のようになります。

function App() {
  return (
    <div className="App">
      <header className="App-header">
        <img src={logo} className="App-logo" alt="logo" />
        <p>
          Hello, World!
        </p>
      </header>
    </div>
  );
}

ステートメントのエクスポート

App.js ファイルの一番下にある export default App は、App コンポーネントを他のモジュールで使用できるようにしています。

index を詳しくみる

src/index.js を開きましょう。App コンポーネントを使用しているのがわかります。このファイルはアプリのエントリーポイントであり、最初は次のようになっています。 

import React from 'react';
import ReactDOM from 'react-dom';
import './index.css';
import App from './App';
import * as serviceWorker from './serviceWorker';

ReactDOM.render(<App />, document.getElementById('root'));

// If you want your app to work offline and load faster, you can change
// unregister() to register() below. Note this comes with some pitfalls.
// Learn more about service workers: https://bit.ly/CRA-PWA
serviceWorker.unregister();

App.js と同様に、ファイルは、実行する必要のあるすべての JS モジュールとその他のアセットをインポートすることから始まります。src/index.css ではアプリ全体に適用されるグローバルスタイルを保持します。ここに、インポートされたAppコンポーネントも確認できます。下部の export により、App.js がインポートできるようになりました。

7 行目 ReactDOM.render() は 2 つの引数で React の関数を呼び出します：

  • <App />: この場合、レンダリングしたいコンポーネント。
  • 内部にコンポーネントをレンダリングする DOM 要素: この場合、root 要素。public/index.html の内側を見ると、これがの <div> であり、<body> すぐ内側の要素であることがわかります。

これらはすべて、App コンポーネントを最初のコンポーネントとして React アプリケーションをレンダリングすることを React に伝えます。

注: JSX では、React コンポーネントと HTML 要素には閉じスラッシュが必要です。そのまま <App> または <img> と書くとエラーが発生します。

Service Worker は、アプリケーションのパフォーマンスを向上させ、ウェブアプリケーションの機能をオフラインで動作させることができる興味深いコードですが、この記事の対象ではありません。5 行目と 9 〜 12 行目を削除します。

最終的なindex.jsファイルは次のようになります。

import React from 'react';
import ReactDOM from 'react-dom';
import './index.css';
import App from './App';

ReactDOM.render(<App />, document.getElementById('root'));

変数と props

次に、JavaScript スキルのいくつかを使用して、コンポーネントの編集と React でのデータの操作をもう少し快適にします。JSX内での変数の使用方法について説明し、データをコンポーネントに渡す方法である props を紹介します (その後、変数を使用してアクセスできます)。

JSXの変数

App.js に戻り、9 行目に注目しましょう。

<img src = {logo} className = "App-logo" alt = "logo" />

ここでは、<img /> タグの src 属性値は中括弧で囲まれています。これは、JSX が変数を認識する方法です。React が {logo} を表示し、アプリの2行目のインポートを参照していることを確認してから、logo ファイルを取得してレンダリングします。

独自の変数を作ってみましょう。App の return ステートメントの前に const subject = 'React'; を追加します。すると、App コンポーネントは次のようになります。 

function App() {
  const subject = "React";
  return (
    <div className="App">
      <header className="App-header">
        <img src={logo} className="App-logo" alt="logo" />
        <p>
          Hello, World!
        </p>
      </header>
    </div>
  );
}

次のように、8 行目を変更して「world」という単語の代わりに subject 変数を使用します。 

function App() {
  const subject = "React";
  return (
    <div className="App">
      <header className="App-header">
        <img src={logo} className="App-logo" alt="logo" />
        <p>
          Hello, {subject}!
        </p>
      </header>
    </div>
  );
}

保存すると、「Hello、world！」の代わりにブラウザーに「Hello、React！」と表示されます。

変数は便利ですが、設定したばかりの変数はReactの機能をあまり活用していません。そこで、propsの登場です。

コンポーネントのprops

props はコンポーネントに渡される任意のデータです。prop はコンポーネント内に記述され、HTML属性と同じ構文を使用します — prop="value"。index.jsを開き、<App/> に最初の prop を渡しましょう。

<App/> コンポーネントに subject という prop を追加して、値は Clarice とします。完了すると、コードは次のようになります。 

ReactDOM.render(<App subject="Clarice" />, document.getElementById('root'));

App.js に戻り、App 関数自体をもう一度見てみましょう。この関数は次のようになっています (簡潔にするために return で短くしています)。 

function App() {
  const subject = "React";
  return (
    // return statement
  );
}

App 関数の引数を変更して、props をパラメーターとして受け入れられるようにします。他のパラメーターと同様 props に、console.log() を入力して、ブラウザーのコンソールにそれを表示させます。すると、次のようになります。 

function App(props) {
  const subject = "React";
  console.log(props);
  return (
    // return statement
  );
}

ファイルを保存し、ブラウザーの JavaScript コンソールを確認します。次のようなログが表示されるはずです。 

Object { subject: "Clarice" }

オブジェクトプロパティ subject は、<App /> コンポーネント呼び出しに追加した subject プロパティに対応し、文字列 Clarice はその値に対応します。React のコンポーネントプロップは、常にこのようにオブジェクトに集約されます。

subject は props の一部です。App.js でこれを使ってみましょう。subject 定数を変更して、React という文字列として定義する代わりに props.subject の値を読み取るようにします。console.log() は必要に応じて削除することもできます。 

function App(props) {
  const subject = props.subject;
  return (
    // return statement
  );
}

保存すると、アプリは「Hello、Clarice！」と挨拶するはずです。index.js に戻り、subject の値を編集して保存すると、テキストが変更されます。

まとめ

これで、React をローカルでインストールする方法、スターターアプリを作成する方法、基本的にどのように機能するかなど、React の入門が終わりました。次の記事では、最初に適したアプリケーションである todo リストの作成を開始します。ただし、その前に、私たちが学んだことのいくつかをおさらいしましょう。

React では:

  • コンポーネントは必要なモジュールをインポートでき、ファイルの下部でエクスポートする必要があります。
  • コンポーネント関数は PascalCase で名前が付けられます。
  • JSX 変数は、{so} のように中括弧で囲むことで読み取ることができます。
  • 一部の JSX 属性は HTML 属性とは異なるため、JavaScript の予約語と競合しません。例えば class、HTML では className JSX に変換されます。複数単語の属性はキャメルケースであることに注意してください。
  • props は、コンポーネント呼び出し内の属性と同じように記述され、コンポーネントに渡されます。
