本ドキュメント郡では、MDN上でデモ、コードサンプル、操作可能な作例等を記述する際のコーディングガイドラインについて概要を説明します。

コードサンプルを記述する際のガイドラインをお探しでしたら、ここがその正しい場所です。ガイドラインを守ることの最大のメリットは、MDN上のサンプルやデモに一貫したスタイルを維持できることです。これにより、ドキュメントの読みやすさや理解のしやすさが底上げされることでしょう。

注: コードの記述方法ではなく、MDN上でのコードの見た目(スタイリング)自体に関する情報をお探しの場合は 執筆スタイルガイドを参照してください。

本ドキュメント群の構成

本ドキュメント郡では、MDN上でコードサンプルを記述る際の一般的なベストプラクティスすについて述べています。ドキュメント群には、以下サブドキュメントがひも付きます：

  • すべてのコードの全般的なガイドライン — 例を表示するための文法的並びにスタイリングのためのガイドライン
  • HTML ガイドライン
  • CSS ガイドライン
  • JavaScript ガイドライン
  • Shell prompt ガイドライン

一般的なベストプラクティス

本セクションでは、特定の機能や関数を例示するため、理解可能かつ必要最低限のサンプルコードを記述するための一般的なベストプラクティスについて説明します。

良いコードサンプルとは：

  • 十分にシンプルで理解しやすい
  • 一方で、興味深く、有用である機能を果たすだけの複雑性を持ち合わせている。

次のことを常に念頭に置くと良いでしょう: 読者はおそらくこのサンプルコードをコピーし、自分のコードに貼り付け、もしかしたら製品化するかもしれません。

そのため、サンプルコードは有用で、一般的に受け入れられているベストプラクティスに従っていなければいけません。また、アプリケーションに脆弱性ををもたらしたり、著しく非効率化したり、冗長化したり、アクセシビリティを阻んだりすることは一切してはいけません。コードサンプルが実行不可能であったり製品化可能なクオリティを満たさない場合、コード内のコメント及び本文内に注意書きを含めます。完全なサンプルではなくその一部分(スニペット)である場合も、そのことを明記します。つまり、そのサンプルコードを実行するための情報は、依存性やセットアップ方法も含めすべて提示しなければなりません。

コードサンプル可能な限り自己完結し、理解しやすいように記述します。プロが唸るような、効率的で賢く、有用なコードを記述することが目的ではなく、すぐに理解ができる洗練されたコードを記述することが理想です。

ガイドライン:
 
  • サンプルコードは短く、理想的には本文で直接扱っている機能についてのみ記述します。
  • サンプルに必要なコードのみ含めます。直接関係のないコードが大量に含まれていると、読者を混乱させ、本筋から気がそれてしまいます。より長く完全なサンプルを提供したい場合は、MDNの Github レポジトリのどれかに格納すると良いでしょう。 (JSBin、 Codepenなどのサービスに保存することも可能です。) そのうえで、完全版のコードへのリンクをサンプルの前後に掲載します。
  • 直接関係のないサーバーサイドのコード、ライブラリ、フレームワーク、 プリプロセッサやその他依存性については含めません。これらはコードの汎用性を阻み、理解や実行を困難にします。可能な場合はバニラコードを使用します。
  • 読者が特定のライブラリ、フレームワーク、プリプロセッサ、その他外部機能に関する知識を持っていることを想定してはいけません。たとえば、BEMやBootstrapのユーザーにのみ理解できるようなクラス名は避け、そのサンプル内での意味が成り立つクラス名にします。
  • たとえより効率的な記述方法がある場合でも、きれいに理解しやすい記述方法を選択します。
  • 簡潔さのために、悪い癖のある記述はしてはいけません。 ( {{HTMLElement("big")}} や {{domxref("Document.write", "document.write()")}}); のような表現は避け、正しい記述を行いましょう。)
  • APIのデモで複数のAPIを利用している場合、どのAPIを呼び出していて、どのAPIがどの機能を提供しているか明記します。
